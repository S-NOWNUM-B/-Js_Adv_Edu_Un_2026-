## 2. Конкретные промпты

### Промпт 1:
"Объясни три семейства паттернов проектирования GoF: Creational, Structural и Behavioral"

Что получил: Подробное объяснение каждого семейства паттернов, их назначения и отличий. Примеры типовых задач, которые решает каждое семейство.

### Промпт 2:
"Как идентифицировать паттерн Singleton в коде JavaScript"

Что получил: Ключевые признаки паттерна Singleton: проверка существующего экземпляра, статическое свойство _instance, возврат существующего объекта.

### Промпт 3:
"В чем разница между паттернами Facade и Mediator"

Что получил: Facade упрощает интерфейс подсистем (структурный), Mediator координирует взаимодействие между объектами (поведенческий). Разные семейства для разных задач.

### Промпт 4:
"Что означает различие между Class-based и Object-based паттернами в GoF"

Что получил: Class-based используют наследование (Factory Method), Object-based используют композицию (Singleton, Observer). Отношения определяются на разных этапах.

### Промпт 5:
"Почему важна правильная категоризация паттернов проектирования"

Что получил: Категоризация помогает сузить область поиска решения. Неправильный выбор семейства может привести к антипаттернам, таким как God Object.

## 3. Модификации и проверки
- Анализ 10 сниппетов кода для идентификации паттернов
- Проверка соответствия выбранных паттернов признакам из GoF
- Валидация классификации по семействам Creational/Structural/Behavioral
- Сопоставление паттернов с уровнями Class/Object

## 4. Что узнал
1. Как различать три основных семейства паттернов GoF по их назначению
2. Ключевые признаки 10 классических паттернов проектирования
3. Разницу между Class-based и Object-based подходами
4. Почему категоризация критична для избежания архитектурных ошибок
5. Как идентифицировать паттерны по структуре кода и используемым методам

# AI_report lab_6.2

## 1. Информация о использовании AI
- Инструмент: Copilot / GitHub, Gemini / Google AI Studio
- Дата: 19.02.2026

## 2. Конкретные промпты

### Промпт 1:
"Какой паттерн лучше использовать для создания единственного экземпляра конфигурации приложения"

Что получил: Рекомендацию использовать Singleton с обоснованием: глобальная точка доступа, предотвращение дублирования, контроль создания.

### Промпт 2:
"Как реализовать паттерн Adapter для интеграции старого API с новым интерфейсом"

Что получил: Структуру Adapter класса: оборачивание старого объекта, преобразование интерфейса, делегирование вызовов к legacy-коду.

### Промпт 3:
"В чем разница между паттернами Observer и Mediator для системы уведомлений"

Что получил: Observer для независимых компонентов, подписывающихся на события. Mediator для централизованной координации взаимодействия.

### Промпт 4:
"Когда использовать Builder вместо обычного конструктора"

Что получил: Builder подходит для сложных объектов с множеством шагов создания и различными представлениями (HTML/PDF отчёты).

### Промпт 5:
"Как паттерн Flyweight помогает оптимизировать использование памяти"

Что получил: Разделение intrinsic (общего) и extrinsic (уникального) состояния. Кэширование повторяющихся данных для экономии памяти.

### Промпт 6:
"Найди ошибки в реализации паттерна Singleton в JavaScript"

Что получил: Выявление проблем с инициализацией, проверка на корректное сохранение экземпляра, правильное использование статических свойств.

## 3. Модификации и проверки
- Выбор подходящих паттернов для 6 различных сценариев
- Обоснование выбора семейства паттерна для каждой проблемы
- Реализация Singleton для системы конфигурации
- Реализация Adapter для интеграции legacy payment API
- Тестирование работоспособности реализованных паттернов

## 4. Что узнал
1. Как анализировать проблему и определять подходящее семейство паттернов
2. Критерии выбора конкретного паттерна внутри семейства
3. Практическую реализацию Singleton с проверкой единственности экземпляра
4. Как использовать Adapter для обеспечения совместимости интерфейсов
5. Компромиссы между различными решениями (Observer vs Mediator, Factory vs Builder)
