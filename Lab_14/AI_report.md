# AI_report lab_14.1

## 1. Информация о использовании AI
- Инструмент: Copilot / GitHub, Gemini / Google AI Studio
- Дата: 16.04.2026

## 2. Конкретные промпты

### Промпт 1:
"Как построить минимальный SSR сервер на Node.js без фреймворков"

Что получил: Шаблон `http.createServer`, маршрутизация по `req.url`, возврат HTML и клиентского JavaScript как отдельных ответов.

### Промпт 2:
"Как передать серверные данные в клиент для гидрации"

Что получил: Паттерн инъекции `window.__INITIAL_STATE__` через `<script>` с `JSON.stringify` для передачи данных пользователя в браузер.

### Промпт 3:
"Как показать разницу SSR и hydration на учебном примере"

Что получил: Сценарий с предварительно отрендеренным профилем, состоянием `loading`, а затем активацией кнопки и обработчиков на клиенте после загрузки.

### Промпт 4:
"Как безопасно выполнять hydration после DOMContentLoaded"

Что получил: Пошаговую схему: проверка целевых DOM-узлов, чтение начального состояния, мягкая деградация при отсутствии элементов и логирование этапов.

## 3. Модификации и проверки
- Реализован SSR endpoint `/` с серверной подгрузкой данных пользователя
- Реализован endpoint `/client.js` для отдачи клиентского кода гидрации
- Добавлена инъекция `window.__INITIAL_STATE__` для передачи состояния с сервера в клиент
- Реализована демонстрационная гидрация: снятие `loading`, разблокировка кнопки и обработчик переключения статуса
- Проверен запуск сервера на `http://localhost:3000`

## 4. Что узнал
1. Как разделяются серверный рендеринг и клиентская гидрация
2. Как передавать состояние SSR-страницы в клиент без повторного запроса
3. Почему FCP может быть быстрым при SSR, а интерактивность появляется позже
4. Как организовать защитные проверки узлов перед гидрацией

# AI_report lab_14.2

## 1. Информация о использовании AI
- Инструмент: Copilot / GitHub, Gemini / Google AI Studio
- Дата: 16.04.2026

## 2. Конкретные промпты

### Промпт 1:
"Как объяснить Islands Architecture на чистом JavaScript"

Что получил: Модель разделения страницы на статический океан и интерактивные острова, где JS загружается только для нужных блоков.

### Промпт 2:
"Как сделать реестр island-компонентов"

Что получил: Паттерн `IslandRegistry` (словарь функций), где ключом выступает тип острова, а значением функция инициализации.

### Промпт 3:
"Как реализовать lazy hydration для островов"

Что получил: Использование `IntersectionObserver` для отложенного монтирования островов, когда они попадают в viewport.

### Промпт 4:
"Какие примеры островов лучше подойдут для лабораторной"

Что получил: Два независимых интерактивных блока: счётчик количества и выбор цвета с визуальным превью.

## 3. Модификации и проверки
- Реализован `IslandRegistry` с двумя островами: `counter` и `color-picker`
- Реализован discovery островов по атрибуту `data-island-type`
- Добавлена ленивая гидрация через `IntersectionObserver` с порогом 10%
- Добавлено отключение наблюдения `unobserve` после инициализации острова
- Проверено, что статический контент работает без обязательной интерактивности

## 4. Что узнал
1. Как islands-подход снижает стоимость клиентского JavaScript
2. Как регистр поведения упрощает масштабирование интерактивных блоков
3. Как `IntersectionObserver` позволяет гидрировать UI по требованию
4. Почему разделение статического и интерактивного контента улучшает производительность
